# docs-site-template

A starter for a gated docs site on Cloudflare. Drop HTML docs into `public/`,
push to `main`, and a Worker serves them behind one of three access models. The
index page is generated from git history plus per-folder metadata, so you never
hand-maintain a list of links.

| | |
|---|---|
| Serves | static HTML/Markdown/PDF/SPA assets from `public/` |
| Gate | a Cloudflare Worker (`src/auth-worker.ts`) |
| Deploy | push to `main` → GitHub Actions runs `wrangler deploy` |
| Index | generated by `scripts/build_index.py` (git + `_meta.json`) |

## Three access models

Pick one per site with the `AUTH_MODE` var in `wrangler.toml`.

| `AUTH_MODE` | Who gets in | Use it for |
|---|---|---|
| `site` | one shared username + password, Turnstile on the login | a team-internal site, single trust boundary |
| `per-doc` | a gated index plus a separate password per document | client deliverables where each client sees only their own doc |
| `none` | nobody is stopped by the Worker | a fully public site, or one you gate upstream with Cloudflare Access |

All three share the same machinery: an HMAC-signed session cookie, optional
Cloudflare Turnstile on the login form, and KV-backed rate limiting that blocks
an IP after repeated failures. Passwords live in Wrangler secrets or KV, never in
the repo and never in client JavaScript.

### `site` mode

One login gates every URL. Set `SITE_USER`, `SITE_PASS`, and `SESSION_SECRET` as
secrets. A correct login plus a passing Turnstile sets a 30-day session cookie.

### `per-doc` mode

The index at `/` is gated by `INDEX_PASSWORD`. Each document folder can carry its
own password stored in KV under `pw:<slug>`. A client given the password for
`/acme-deck/` unlocks only that path and never sees the index or other clients'
docs. A folder with no KV password falls back to `INDEX_PASSWORD`, so nothing is
ever served ungated. If `INDEX_PASSWORD` is unset, the Worker returns 503 rather
than open the site, so a misconfiguration fails closed.

### `none` mode

The Worker adds security headers and serves the assets. Reach for this when the
site is public, or when you put Cloudflare Access in front of the hostname and let
Access handle identity (the SSO/email-gated path). See the cloudflare skill for
wiring Access on a zone.

## Cloudflare Access (the SSO-gated path)

`none` mode plus a Cloudflare Access application on the custom domain gives you
email/SSO gating without a password in the repo. Create the Access app and policy
against the zone with a token scoped for Access, point it at the hostname, and the
Worker stays a passthrough. This is the right model for an internal site where you
want per-identity audit instead of a shared password.

## Setup

1. Copy this folder into a new repo and replace the `replace-*` placeholders in
   `wrangler.toml` (`name`, `account_id`, the route `pattern`).

2. Install and generate a session secret:
   ```bash
   npm install
   sh scripts/setup.sh secrets
   ```
   That sets `SESSION_SECRET` and prints the remaining secrets to set for your
   chosen mode.

3. For `site` mode, set the login:
   ```bash
   npx wrangler secret put SITE_USER
   npx wrangler secret put SITE_PASS
   ```

4. For `per-doc` mode, create the KV namespace and set the index password:
   ```bash
   sh scripts/setup.sh kv          # prints the namespace id
   npx wrangler secret put INDEX_PASSWORD
   sh scripts/setup.sh set-doc-password acme-deck 'their-password'
   ```
   Paste the KV id into `wrangler.toml` under a `[[kv_namespaces]]` block with
   `binding = "AUTH_KV"`. KV also backs rate limiting in every mode, so add it
   even for `site` mode if you want the IP lockout.

5. To enable Turnstile, create a widget (READALL token in the cloudflare skill),
   set `TURNSTILE_SITE_KEY` under `[vars]` in `wrangler.toml`, and set the secret:
   ```bash
   npx wrangler secret put TURNSTILE_SECRET_KEY
   ```
   With no Turnstile keys, the login still works; it just skips the captcha.

6. Activate the git hooks and deploy:
   ```bash
   sh .githooks/install.sh
   git push
   ```

The GitHub Actions workflow needs a `CLOUDFLARE_API_TOKEN` repo secret with
Workers deploy scope.

## Adding a doc

The `public/index.html` listing page is generated. Never hand-edit it.

1. Create `public/<slug>/index.html`.
2. Add `public/<slug>/_meta.json`:
   ```json
   { "title": "...", "desc": "...", "confidential": false, "tags": ["Doc"] }
   ```
   Tags must be in `scripts/tags.json`. Add a new one there first if needed.
3. Rebuild and push:
   ```bash
   python3 scripts/build_index.py
   git add -A && git commit -m "Add <slug>" && git push
   ```

`pre-commit` regenerates and stages the index when a commit touches `public/`.
`pre-push` and the CI `--check` step block a stale or off-vocabulary index.

## Branding the index

Edit `scripts/site.json` for the stamp, heading, sub-line, and footer. Edit
`scripts/index_template.html` for the full look. Rebuild to apply.

## What the Worker does not protect against

- A shared password being passed around outside the trusted group. Rotate it.
- Cloudflare reading content at the edge. This is any CF site.
- Turnstile is bot friction, not identity. For per-identity audit use Access.

## Files

```
src/auth-worker.ts        the gate (site / per-doc / none)
public/                   your docs + the generated index.html
scripts/build_index.py    index generator (git + _meta.json)
scripts/site.json         index branding
scripts/tags.json         allowed tag vocabulary
scripts/setup.sh          secrets / KV / per-doc password helper
.githooks/                index enforcement (install.sh once per clone)
.github/workflows/        wrangler deploy on push to main
wrangler.toml             worker + assets + AUTH_MODE
```
